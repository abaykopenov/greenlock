# Руководство пользователя Greenlock

**Greenlock** — это детерминированный гейт верификации (verify-gate) для изменений кода, созданных искусственным интеллектом (или человеком). Его главная цель — гарантировать, что ни одно ошибочное или регрессионное изменение не попадет в основную ветку репозитория (**WRONG-APPLY = 0**).

---

## 1. Философия и архитектура

Greenlock **не является** очередным генератором кода. Это страховой слой безопасности, который встраивается *между* разработчиком (или ИИ-агентом вроде Devin, Copilot, Cursor) и репозиторием.

Когда поступает патч (в формате unified diff):

```
patch (unified diff)  ─►  Создание песочницы (sandbox)
                           ├─ Применение патча (git apply)
                           ├─ Danger-check: проверка на опасные конструкции (eval, os.system)
                           ├─ Closed-world: отсутствие ссылок на несуществующие символы
                           ├─ Запуск тестов репозитория (Test Oracle)
                           └─ Сравнение результатов с базовой линией (Baseline) → Регрессия?
                                                     │
                           Успешно + нет регрессии ──►  ✅ MERGE (Код принят)
                           Любая ошибка            ──►  🛑 REJECT (Код отклонен с причиной)
```

## 2. Быстрый старт: Интерактивный `greenlock init`

Самый простой способ настроить Greenlock в вашем проекте — использовать интерактивный мастер инициализации. 

### Шаг 1. Установка пакета
Склонируйте проект и установите его в режиме разработки:
```bash
git clone https://github.com/abaykopenov/greenlock.git
cd greenlock
pip install -e ".[all]"
```

### Шаг 2. Инициализация в целевом проекте
Перейдите в каталог вашего проекта (где настроен Git) и запустите команду инициализации:
```bash
python3 -m greenlock.gate init
```

### Шаг 3. Интерактивная настройка
Мастер запустит опрос в терминале:
1. **Выбор верификатора**: укажите тип вашего проекта:
   * `1` для Python (`pytest`)
   * `2` для Node.js (`node`)
   * `3` для Go (`go test`)
   * `4` для Rust (`cargo test`)
   * `5` для кастомного запуска (`custom`)
   * `Enter` для автоматического определения по файлам-манифестам.
2. **Команда тестирования**: введите команду (например, `pytest --maxfail=1`).
3. **Команда проверки синтаксиса** (только для `custom` верификатора).

### Что произойдет под капотом:
* В корне вашего проекта будет создан конфигурационный файл `greenlock.json` с вашими настройками.
* В папку `.git/hooks/pre-commit` запишется bash-скрипт хука. С этого момента при каждом вызове `git commit` изменения будут автоматически проверяться через Greenlock. Если проверка провалится, коммит будет заблокирован.

## 3. Конфигурация через `greenlock.json`

Файл `greenlock.json` позволяет явно управлять поведением Greenlock в вашем репозитории. Пример конфигурации:

```json
{
  "verifier": "pytest",
  "test_command": "python3 -m pytest --maxfail=1 --ignore=tests/integration"
}
```

### Доступные параметры:
* `"verifier"`: тип верификатора (`"pytest"`, `"node"`, `"go"`, `"rust"`, `"custom"`).
* `"test_command"`: переопределение команды запуска тестов.
* `"syntax_command"`: команда статической проверки синтаксиса (используется только для верификатора `"custom"`).

## 4. Безопасность и Docker-изоляция (RCE Mitigation)

Поскольку Greenlock выполняет тесты проверяемого проекта, запуск недоверенного кода (например, написанного сторонним ИИ-агентом) несет риск удаленного выполнения кода (RCE) на вашей хост-машине.

### Включение Docker-изоляции
Для защиты хост-системы настройте выполнение тестов внутри изолированных контейнеров:

1. **Соберите образ**:
   ```bash
   docker build -t greenlock:latest .
   ```
2. **Включите изоляцию** в конфигурационном файле `greenlock.local.json` в корне Greenlock:
   ```json
   {
     "docker": "true"
   }
   ```
   *Или через переменную окружения:* `export GREENLOCK_DOCKER=true`

### Как это работает:
При запуске тестов Greenlock автоматически создаст временный контейнер:
* Сеть полностью отключена (`--network none`) — злоумышленник не сможет отправить ваши секреты или данные вовне.
* Корневая файловая система примонтирована только для чтения (`--read-only`), а песочница проекта смонтирована как `ro`.
* Вся запись разрешена только во временную директорию `tmpfs`.
* Лимиты на память, CPU и лимит PID предотвращают fork-бомбы.

## 5. Способы использования

### Сценарий А: Проверка локального диффа в терминале
Вы можете проверить любые изменения перед фиксацией вручную:
```bash
# Передать готовый файл диффа:
python3 -m greenlock.gate /путь/к/репо change.diff

# Или передать дифф текущих изменений из git напрямую:
git diff | python3 -m greenlock.gate /путь/к/репо -
```

### Сценарий Б: Интеграция с ИИ-агентами (Claude Code, Cursor) через MCP
Greenlock поддерживает протокол Model Context Protocol (MCP). Вы можете подключить его как сервер инструментов к вашему ИИ-редактору:
```bash
pip install "greenlock[mcp]"
greenlock-mcp
```
Агент получит доступ к инструментам:
* `verify_patch` — проверить предложенный диф кодовой базы.
* `generate_characterization_tests` — сгенерировать характеризационные тесты для покрытия легаси-кода.
* `harden_and_verify` — автоматически покрыть тестами измененные файлы и выполнить гейт-проверку.

### Сценарий В: Интеграция в GitHub Actions
Блокируйте слияние PR, если они нарушают целостность кодовой базы. Добавьте в workflow вашего проекта:
```yaml
- uses: actions/checkout@v4
  with: { fetch-depth: 0 }
- uses: actions/setup-python@v5
  with: { python-version: "3.11" }
- run: pip install -r requirements.txt # Установите зависимости проекта
- uses: abaykopenov/greenlock@main   # Запуск гейта Greenlock
```
Если верификация вернет `REJECT`, шаг завершится с ошибкой, и GitHub заблокирует кнопку Merge.
